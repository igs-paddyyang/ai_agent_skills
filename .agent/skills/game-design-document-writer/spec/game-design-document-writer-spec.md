# game-design-document-writer Skill Spec

**作者**: paddyyang
**日期**: 2026-03-19
**版本**: v0.1
**目標**: 作為 skill-creator 的輸入，產出 game-design-document-writer 技能

---

## 1. 技能定義

- **名稱**: game-design-document-writer
- **一句話描述**: 根據遊戲構想產出結構化的遊戲企劃文件（GDD），涵蓋概念、玩法、敘事、美術、音效、技術與營運規劃
- **觸發描述（description 欄位草稿）**:
  > 根據遊戲構想或需求描述，產出完整的遊戲企劃文件（Game Design Document, GDD）。涵蓋遊戲概述、核心玩法循環、系統機制、敘事與世界觀、美術風格、音效設計、技術規格、營運與商業模式等章節。支援完整 GDD 和精簡版 One Pager 兩種輸出模式。當使用者提到遊戲企劃、遊戲設計文件、GDD、game design document、遊戲規格書、遊戲提案、遊戲概念文件、寫遊戲企劃、設計遊戲時，請務必使用此技能。

## 2. 行為規格

### 2.1 輸入

- **遊戲構想**（必要）：使用者對遊戲的描述，可以是一句話概念、詳細需求、或參考遊戲
- **輸出模式**（選用）：`full`（完整 GDD，預設）或 `onepager`（精簡版 One Pager）
- **遊戲類型提示**（選用）：手遊、PC、主機、網頁遊戲、博弈遊戲等
- **參考資料**（選用）：競品分析、美術參考、技術限制等

### 2.2 處理邏輯

#### 步驟 1：概念萃取（Understanding Lock）

從使用者輸入中提取或推導以下核心要素，確認後才進入下一步：

- 遊戲名稱（暫定）
- 類型（Genre）與子類型
- 目標平台
- 目標受眾
- 核心賣點（Unique Selling Point）
- 一句話概念（Elevator Pitch）

#### 步驟 2：章節展開

根據輸出模式展開對應章節：

**完整 GDD 模式（full）** — 展開以下 10 個章節：

1. 遊戲概述（Overview）— 名稱、類型、平台、受眾、靈感來源、Elevator Pitch
2. 核心玩法（Core Gameplay）— 核心循環（Core Loop）、主要操作、進程系統、難度曲線
3. 系統機制（Game Systems）— 經濟系統、戰鬥/互動系統、成長系統、AI 行為、物理規則
4. 關卡與內容（Level & Content）— 關卡結構、地圖設計、任務系統、可重玩性
5. 敘事與世界觀（Narrative & World）— 故事大綱、角色設定、世界觀、主題
6. 美術風格（Art & Visual）— 視覺風格、角色設計方向、場景設計方向、UI/UX 設計
7. 音效設計（Audio）— 音樂風格、音效清單、語音需求
8. 技術規格（Technical）— 引擎選擇、平台需求、效能目標、網路架構（若適用）
9. 營運與商業模式（Business & Monetization）— 商業模式、定價策略、營運計畫、KPI 指標
10. 開發計畫（Production Plan）— 里程碑、團隊配置、風險評估

**精簡版 One Pager 模式（onepager）** — 濃縮為單頁：

1. 遊戲名稱 + Elevator Pitch
2. 類型 / 平台 / 受眾
3. 核心玩法循環（圖示）
4. 3-5 個關鍵特色
5. 美術風格參考（文字描述）
6. 商業模式摘要
7. 競品對比（表格）

#### 步驟 3：數值與細節填充

- 為系統機制提供具體的數值範例（如：HP、攻擊力、經濟產出率）
- 數值標註為「建議值，需經實際測試調整」
- 使用表格呈現數值對比（如：角色屬性表、道具效果表）

#### 步驟 4：產出與儲存

- 產出 Markdown 格式的 GDD 文件
- 預設儲存路徑：`docs/<遊戲名稱>-gdd.md`
- One Pager 模式儲存為：`docs/<遊戲名稱>-onepager.md`

### 2.3 輸出格式

產出的 GDD 文件遵循 `references/gdd-template.md` 的結構模板。核心格式要求：

- Markdown 格式，章節使用 `##` 標題
- 表格優先於長段落（提升可掃描性）
- 每個章節都要有實質內容，不要空殼標題
- 數值用表格呈現，標註「建議值」
- 繁體中文撰寫，遊戲術語保留英文
- 文件末尾標註產出技能版本與日期

### 2.4 邊界案例

- **輸入過於模糊**（如「幫我設計一個遊戲」）：引導使用者至少提供類型和核心玩法概念，再開始產出
- **非遊戲需求**：不處理純軟體規格、API 設計等非遊戲企劃需求，引導使用 software-spec-writer
- **數值平衡**：明確標註所有數值為建議值，需經實際測試調整，不保證平衡性
- **美術資源**：僅提供文字描述的美術方向，不產出實際圖片或概念圖

## 4. 測試案例

| # | 提示詞 | 預期輸出 | 驗證方式 |
|---|--------|---------|---------|
| 1 | 「幫我設計一個末日殭屍捕魚機博弈遊戲」 | 完整 GDD，包含 10 章節，博弈類型側重經濟系統與 RTP 數值 | 檢查 10 章節完整性、數值表格存在、博弈特有欄位（RTP、賠率表） |
| 2 | 「設計一個手機休閒消除遊戲，onepager 模式」 | 精簡版 One Pager，單頁涵蓋核心概念、玩法循環、商業模式 | 檢查 7 個 One Pager 區塊完整性、篇幅控制在合理範圍 |
| 3 | 「幫我寫一個開放世界 RPG 的 GDD，參考薩爾達和艾爾登法環」 | 完整 GDD，RPG 類型側重成長系統、敘事、關卡設計 | 檢查 RPG 特有章節深度（角色成長、技能樹、任務系統）、靈感來源引用 |
| 4 | 「幫我設計一個遊戲」（模糊輸入） | 引導使用者提供更多資訊（類型、平台、核心概念），不直接產出 | 檢查是否有引導問題而非直接產出空泛 GDD |
| 5 | 「幫我寫一個 API 設計文件」（非遊戲需求） | 拒絕處理，引導使用 software-spec-writer | 檢查是否正確識別為非遊戲需求並引導 |

## 5. 備註

- 本技能是先前 `game-copywriting`（已移除）的升級替代品，從「文案撰寫」提升為「完整企劃文件產出」
- 參考來源：
  - antigravity-awesome-skills 的 Business 分類（copywriting 模式）
  - SkillsMP 上的 game-development 相關技能（vudovn/antigravity-kit）
  - LobeHub 上的 Game Design Doc Creator（sunholo-data）
  - Indie Game Academy 的 GDD 模板結構
  - 業界標準 GDD 結構（gamedeveloper.com、nuclino.com）
- 技能結構模式：工作流程型（步驟 1 概念萃取 → 步驟 2 章節展開 → 步驟 3 數值填充 → 步驟 4 產出）
- `genre-guides.md` 的類型指南讓同一個技能能適應不同遊戲類型，避免為每種類型建立獨立技能
- 後續可考慮加入 `assets/` 目錄放置視覺化的 Core Loop 流程圖範本
