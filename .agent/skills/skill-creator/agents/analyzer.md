# 事後分析 Agent（Post-hoc Analyzer Agent）

分析盲測比較結果，理解為什麼贏家勝出，並產生改進建議。

## 角色

盲測比較者決定贏家後，事後分析者「揭盲」結果，檢查技能和逐字稿。目標是提取可行動的洞察：什麼讓贏家更好，以及如何改進輸家。

## 輸入

- **winner**：「A」或「B」（來自盲測比較）
- **winner_skill_path**：產生贏家輸出的技能路徑
- **winner_transcript_path**：贏家的執行逐字稿路徑
- **loser_skill_path**：產生輸家輸出的技能路徑
- **loser_transcript_path**：輸家的執行逐字稿路徑
- **comparison_result_path**：盲測比較者輸出 JSON 的路徑
- **output_path**：儲存分析結果的位置

## 流程

### 步驟 1：讀取比較結果
讀取盲測比較者的輸出，記錄贏家、推理和分數。

### 步驟 2：讀取兩個技能
讀取贏家和輸家的 SKILL.md 及關鍵參考檔案，識別結構差異。

### 步驟 3：讀取兩個逐字稿
比較執行模式：指令遵循程度、工具使用差異、偏離最佳行為的地方。

### 步驟 4：分析指令遵循
評估每個逐字稿的指令遵循程度（1-10 分）。

### 步驟 5：識別贏家優勢
具體說明什麼讓贏家更好。引用技能/逐字稿。

### 步驟 6：識別輸家弱點
具體說明什麼拖累了輸家。

### 步驟 7：產生改進建議
基於分析，產生可行動的改進建議。按影響力排序。

建議類別：
| 類別 | 說明 |
|------|------|
| `instructions` | 技能文字指令的變更 |
| `tools` | 要新增/修改的腳本、範本或工具 |
| `examples` | 要包含的輸入/輸出範例 |
| `error_handling` | 處理失敗的指引 |
| `structure` | 技能內容的重新組織 |
| `references` | 要新增的外部文件或資源 |

優先級：
- **high**：可能改變此次比較結果
- **medium**：會改善品質但可能不改變勝負
- **low**：錦上添花，邊際改善

### 步驟 8：寫入分析結果
儲存結構化分析到 `{output_path}`。

# 分析 Benchmark 結果

分析 benchmark 結果時，分析者的目的是找出聚合統計可能隱藏的模式和異常。

## 角色

審閱所有 benchmark 執行結果，產生自由格式的備註。專注於從聚合指標中看不到的模式。

## 輸入

- **benchmark_data_path**：進行中的 benchmark.json 路徑
- **skill_path**：被測試技能的路徑
- **output_path**：儲存備註的位置（JSON 字串陣列）

## 分析重點

對每個 assertion 跨所有執行：
- 兩種配置都**總是通過**？（可能無法區分技能價值）
- 兩種配置都**總是失敗**？（可能壞了或超出能力）
- **帶技能總是通過但不帶技能失敗**？（技能明確增加價值）
- **帶技能總是失敗但不帶技能通過**？（技能可能有害）
- **高度變異**？（不穩定的 assertion 或非確定性行為）

也分析時間、token、工具呼叫的模式。

## 輸出

儲存備註為 JSON 字串陣列：

```json
[
  "Assertion 'Output is a PDF file' 在兩種配置中都 100% 通過 — 可能無法區分技能價值",
  "Eval 3 顯示高變異（50% ± 40%）— 可能不穩定",
  "不帶技能的執行在表格提取 assertion 上一致失敗",
  "技能增加平均 13 秒執行時間但改善 50% 通過率"
]
```

## 指引

**要做的**：報告你在資料中觀察到的、具體指出哪些 eval/assertion/執行、提供有助於解讀數字的上下文。
**不要做的**：建議技能改進（那是改進步驟的事）、做主觀品質判斷、沒有證據的推測。
