# 技能規格撰寫師（Skill Spec Writer）

為 Kiro Skill 產出輕量、結構化的規格文件，讓 skill-creator 能直接消費並進入檔案產生階段。

## 核心理念

software-spec-writer 服務專案級規格（多模組、多 Phase），而 skill-spec-writer 服務技能級規格（單一觸發、單一行為）。兩者互補，不重疊：

```
software-spec-writer → 專案級規格（如 arkbot-spec.md）
skill-spec-writer    → 技能級規格（餵給 skill-creator）
```

產出的 Skill Spec 對齊 skill-creator 階段 1（意圖捕捉）的 4 個核心問題，加上結構化的行為定義與測試案例，讓 skill-creator 拿到後可跳過來回問答，直接進入階段 3（檔案產生）。

---

## 輸入

接受以下內容（至少提供第 1 項）：

1. **技能構想**：使用者想讓 AI 做什麼（必要）
2. **觸發情境**：什麼時候該觸發這個技能（選用，可協助推導）
3. **參考資料**：現有文件、API 規格、範例輸出等（選用）
4. **約束條件**：技術限制、輸出格式要求、依賴項（選用）

---

## 工作流程

### 階段 1：意圖釐清（20%）

```
[████░░░░░░░░░░░░░░] 20% - 步驟 1/5：意圖釐清
```

從使用者的描述中提取或確認以下 4 個核心問題：

1. **這個技能要讓 AI 做什麼？** — 核心行為的一句話描述
2. **何時應該觸發？** — 使用者短語、情境、檔案類型
3. **預期的輸出格式是什麼？** — 產出物的結構與形式
4. **是否需要測試案例？** — 有客觀可驗證輸出的技能適合；主觀輸出的通常不需要

如果使用者的描述已經足夠清楚，不需要逐一追問，直接整理後確認即可。避免不必要的來回。

### 階段 2：行為定義（40%）

```
[████████░░░░░░░░░░] 40% - 步驟 2/5：行為定義
```

根據意圖釐清的結果，展開完整的行為規格：

- **輸入規格**：接受什麼類型的輸入、必要 vs 選用欄位
- **處理邏輯**：核心演算法或決策流程（用簡潔的步驟描述，不需要寫程式碼）
- **輸出規格**：產出物的結構、格式、範例
- **邊界案例**：不該做什麼、錯誤處理方式
- **觸發描述草稿**：為 SKILL.md 的 description 欄位草擬觸發文字

觸發描述的撰寫原則（對齊 skill-creator 規範）：
- 同時包含功能描述和使用情境關鍵字
- 稍微「積極」一些，對抗觸發不足的傾向
- 長度 ≤ 1024 字元

### 階段 3：資源規劃（60%）

```
[████████████░░░░░░] 60% - 步驟 3/5：資源規劃
```

評估技能是否需要附帶資源，以及各資源的用途：

- **scripts/**：是否有確定性/重複性任務需要腳本化？列出每個腳本的名稱與用途
- **references/**：是否有大量指南或規範需要按需載入？列出每份文件的名稱與涵蓋範圍
- **assets/**：是否需要範本、圖片等輸出用檔案？列出每個檔案的名稱與用途

不需要的資源類別直接省略，不要為了結構完整而硬塞空目錄。

同時評估 SKILL.md 的預估行數。如果核心指令超過 500 行，規劃哪些內容應拆到 references/ 按需載入。

### 階段 4：測試案例設計（80%）

```
[████████████████░░] 80% - 步驟 4/5：測試案例
```

如果階段 1 判定需要測試案例，設計 3-5 個覆蓋核心場景的測試：

每個測試案例包含：
- **提示詞**：模擬使用者的真實輸入
- **預期輸出**：描述期望的結果（不需要精確到逐字，描述結構和關鍵內容即可）
- **驗證方式**：如何判斷輸出是否正確

測試案例應覆蓋：
- 典型使用場景（happy path）
- 邊界案例（edge case）
- 不該觸發的場景（negative case，選用）

如果技能的輸出是主觀的（如文案撰寫），可以改用定性評估標準替代精確的預期輸出。

### 階段 5：產出 Skill Spec（100%）

```
[████████████████████] 100% - 步驟 5/5：產出規格
```

將前 4 個階段的成果整合為一份結構化的 Skill Spec 文件。

## 設計原則

- **輕量優先**：Skill Spec 不是專案規格，不需要架構圖、目錄結構、Phase 分解、時間估算。聚焦在「技能做什麼」和「怎麼驗證」
- **對齊消費者**：產出格式直接對應 skill-creator 階段 1 的 4 個問題 + 階段 3 需要的資訊
- **一份文件**：所有資訊集中在一份 Skill Spec，不拆分多個文件
- **可跳過的階段**：如果使用者已經很清楚技能要做什麼，可以跳過意圖釐清直接進入行為定義

## 自我檢核

完成 Skill Spec 後，確認：
1. 技能定義是否回答了 skill-creator 的 4 個核心問題
2. 觸發描述是否同時包含功能描述和情境關鍵字
3. 輸出格式是否具體到 skill-creator 能直接實作
4. 測試案例是否覆蓋 happy path 和 edge case
5. 資源需求是否合理（不多不少）
6. 整份 Spec 是否能讓 skill-creator 跳過階段 1 直接進入階段 3

## 參考資源

產出 Skill Spec 時，讀取 `references/skill-standards.md` 以參考：
- Agent Skills 規範（agentskills.io）— frontmatter 欄位、命名規則、漸進式揭露
- 官方技能範例（anthropics/skills）— 3 種結構模式的實際寫法
- 社群技能範例（antigravity-awesome-skills）— 9 大分類體系、2 種進階結構模式
- description 撰寫公式與範例
- 4 種技能結構模式（參考型 / 工具型 / 工作流程型 / 選擇性閱讀型）

在階段 2（行為定義）撰寫觸發描述時，參考 §5 的 description 撰寫範例。
在階段 3（資源規劃）評估結構時，參考 §4 的技能結構模式總結。

## 與其他技能的關係

```
skill-spec-writer    → 產出 Skill Spec（技能級規格）
  ↓
skill-creator        → 消費 Skill Spec，產出 Skill Package
  ↓
deploy to .kiro/skills/ 或 arkbot/skills/
```

software-spec-writer 負責專案級規格，skill-spec-writer 負責技能級規格。如果使用者的需求是「建立一個完整專案」，應引導使用 software-spec-writer；如果是「建立一個技能」，使用本技能。
