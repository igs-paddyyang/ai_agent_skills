# 網頁搜尋摘要師（WebSearch Summarizer）

搜尋或抓取指定網頁內容，產出結構化的 Markdown 摘要文件。適合快速理解一個專案、產品、文件或技術資源的全貌。

## 使用時機

- 使用者提供 URL，想快速了解內容
- 使用者提供關鍵字，想搜尋後整理結果
- 需要將長篇網頁內容濃縮為可行動的摘要

## 工作流程

### 步驟 1：確認輸入來源

使用者可能提供：

| 輸入類型 | 處理方式 |
|---------|---------|
| 完整 URL | 直接抓取該頁面內容 |
| GitHub Repo URL | 抓取 README，若內容不足再抓取子頁面 |
| 搜尋關鍵字 | 先用 web search 找到相關頁面，再抓取最佳結果 |

若使用者未指定輸出路徑，預設存到 `docs/` 目錄，檔名為 `<主題>-summary.md`。

### 步驟 2：抓取與解析內容

1. 使用 web fetch 工具抓取頁面內容
2. 若首次抓取內容不足（僅導覽列或空白），改用 rendered 模式重試
3. 對於 GitHub Repo：
   - 優先抓取 README.md（raw 格式）
   - 若有 CATALOG 或目錄頁面，一併抓取
   - 必要時抓取 2-3 個關鍵子頁面補充細節
4. 對於一般網頁：
   - 抓取主頁面內容
   - 若內容過短，嘗試抓取相關子頁面

### 步驟 3：產出結構化摘要

依照以下模板結構產出 Markdown 文件。根據內容性質彈性調整區塊，但核心結構保持一致。

#### 摘要文件模板

產出的摘要文件參考 `references/output-template.md` 的結構。核心區塊包含：

1. **標題與元資料** — 來源 URL、版本/日期、授權、整理日期
2. **一句話摘要** — 用一句話概括這個資源是什麼
3. **核心概念** — 2-3 句解釋核心理念或運作方式
4. **分類/功能總覽** — 用表格整理主要分類或功能模組
5. **關鍵細節** — 依內容性質展開重要區塊（安裝方式、API、架構等）
6. **與本專案的關係**（選用）— 如果跟本專案有關聯，說明如何應用
7. **行動建議** — 具體的下一步行動，而非泛泛的「可以參考」

#### 寫作原則

- 用繁體中文撰寫，技術名詞保留英文
- 表格優先於長段落，提升可掃描性
- 每個區塊都要有實質內容，不要空殼標題
- 行動建議要具體可執行，不要泛泛而談
- 保留原始來源的關鍵數據（版本號、數量、日期等）
- 不逐字翻譯，而是重新組織資訊結構

### 步驟 4：儲存與回報

1. 將摘要文件寫入指定路徑（預設 `docs/<主題>-summary.md`）
2. 向使用者回報：
   - 檔案路徑
   - 一句話摘要
   - 抓取了哪些頁面
   - 若有無法抓取的內容，說明原因

## 邊界案例

- **需要登入的頁面** → 說明無法抓取，建議使用者提供內容
- **內容過長（>50KB）** → 分段抓取，聚焦最重要的部分
- **非文字內容（影片、圖片為主）** → 說明限制，僅整理可抓取的文字部分
- **404 或無法連線** → 回報錯誤，建議檢查 URL 或改用搜尋關鍵字
